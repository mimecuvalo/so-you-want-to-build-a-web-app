# patterns of consistency and simplicity

designers and developers have a lot in common when you think about it: they‚Äôre both lazy at heart üòú. they both want to ‚ÄúKeep It Simple, Stupid‚Äù (KISS) and are obsessed about details like tabs vs. spaces (or, for designers, `px` vs `em`, hah). knowing this, you will make your designers infinitely happy if you not only pay attention to the general guidelines they are giving you but also the minute details that make your app come alive.

# styleguide

to do the aforementioned collaboration effectively, start with a styleguide, which is effectively a contract between design and development. it includes things like: color palettes, fonts, widgets (buttons, menus, etc.), and iconography.

here are some recommended tools:

- [**Storybook**](https://storybook.js.org/) ([Chromatic](https://www.chromatic.com/) is a side-car tool that can help gather feedback)
- [**React Styleguidist**](https://react-styleguidist.js.org/)

# accessibility (a11y)

this is an important, yet *so-overlooked-it‚Äôs-not-even-funny* area in web development. at the very least, you shouldn‚Äôt be doing silly things like using `<span>` tags where you should be using `<button>` or `<a>` tags. but there‚Äôs *so much more* to a11y than just using the correct HTML elements, like: adding ARIA attributes to your tags, making sure color contrast is correct, placing elements in a certain tab order, and the list goes on and on.

there‚Äôs so much to cover in this area such that it wouldn‚Äôt all fit in this guide. it is my hope that eventually at least UI toolkits (mentioned in a section below) will become robust enough for the average developer so that at least the widgets are adherent to the guidelines ‚Äî we‚Äôre almost there in that regard. here are some must-read guides on learning the in‚Äôs and out‚Äôs:

- [**Google‚Äôs guide**](https://developers.google.com/web/fundamentals/accessibility/)
- [**Chrome DevTools reference**](https://developers.google.com/web/tools/chrome-devtools/accessibility/reference)

programmatically, there are a couple libraries that you can use to get an automated assessment of your website:

- [**aXe**](https://www.deque.com/axe/)
- [**Google‚Äôs Accessibility Developer Tools**](https://github.com/GoogleChrome/accessibility-developer-tools)

anecdotally, at my previous company, a tool was designed internally to display a11y information on every webpage (so that it wasn‚Äôt just hidden in the DevTools). that‚Äôs how important this is!

finally, if you‚Äôre on a Mac be sure to check out the [VoiceOver Utility](https://www.apple.com/accessibility/mac/vision/) to get a real-world sense of how your app is feeling when navigating via keyboard (although i think it only works on Safari, sadly).

# internationalization (i18n)

somewhat in the same vein as a11y is providing access to your site to the people around the world that don‚Äôt necessarily speak the same language that you do. there‚Äôs a lot that goes into i18n, not just translating words, but also the way numbers are formatted, dates, relative dates and pluralization.

the recommended library here is [**react-intl**](https://github.com/yahoo/react-intl/wiki).

# responsiveness

today‚Äôs devices are part of a Cambrian explosion of which the variety is astounding. as such, you need to make your UI‚Äôs adaptable if you are to survive in this new world. most of this involves using CSS media queries (example: `@media (min-width: 600px)`) to gate certain CSS to only apply when a smaller device is being used. learn more [here](https://developers.google.com/web/fundamentals/design-and-ux/responsive/).


## avoid fixed widths

in your CSS you should tend to avoid boxes that have a set size (with `px`) and use percentages or `em` values. the newer `vw` or `vh` properties which are relative to screen size can also be helpful here.


## fluid layouts

your layouts should have the capability to flow vertically when the width is constrained. in effect, the columns of your website should stack on top of each other instead of next to each other.


## font sizes

take into consideration that your users are on handheld devices and they will have to strain their eyes more.


## image/icon responsiveness

in addition to making it easier to click around your app, you‚Äôll also hopefully save some bandwidth which is more constrained when the user is on the go.


## device toolbar

in the top left of Chrome DevTools, next to the Inspect element icon is a button that lets you see how your site is responsive to different mobile device widths. learn it. love it.

# web components

oof, there is a lot going on here that it can make the head spin. you‚Äôll hear related technologies under the umbrella term of web components, including: [Custom Elements](https://developers.google.com/web/fundamentals/web-components/customelements), [`<template>`](https://www.html5rocks.com/en/tutorials/webcomponents/template/), [Shadow DOM](https://developers.google.com/web/fundamentals/web-components/shadowdom), and [ES modules](https://www.html5rocks.com/en/tutorials/webcomponents/imports/).

web components are a way to create reusable components that are encapsulated in the DOM. cross-browser support is almost here but not quite yet so this technology is still a bit bleeding edge. because of all the ups and downs in this area historically, i‚Äôm *almost* ready to recommend going this route vs. more vanilla widgets.

i think i‚Äôm in love with the *idea* of web components. but for some reason they keep failing to gain traction in the community. i would take the ‚Äúassess‚Äù stance on Web Components: they might be poised to still be the dominant way to write components in the future but we‚Äôre still not there yet.

# typography

a good custom font (i.e. from Google Fonts) can add a lot of flavor to your site. that being said, system fonts are still a great choice, too, and much faster! the choice is yours. that being said though, you should use still probably use the system font stack regardless to avoid invisible text while the custom fonts are loading.


## custom fonts

[**Google Fonts**](https://fonts.google.com/). a wonderful set of great fonts that are free to use! here are some [inspirational font choices](https://femmebot.github.io/google-type/) via Google Fonts to get you thinking about how to choose one.


## system font stack

by defaulting to a system font you gain speed and a look for your site that matches the user‚Äôs operating system. here‚Äôs how to implement it:

```css
body {
  font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", "Roboto", "Oxygen",
      "Ubuntu", "Cantarell", "Fira Sans", "Droid Sans", "Helvetica Neue", sans-serif;
}
code {
  font-family: source-code-pro, Menlo, Monaco, Consolas, "Courier New", monospace;
}
```


## readability

for the human eye, reading a wide swath of text is tricky. an ideal width of 45‚Äì75 characters can be achieved thusly:

```css
p {
  width: clamp(45ch, 50%, 75ch);
}
/* or */
p {
  font-size: clamp(1.5rem, 5vw, 3rem);
}
```
